# 北京天通苑小区房价历史分析报告 — 执行方案与 AI 交互流程

> 本文档是本次「AI 数据报告全栈开发」笔试的总纲，包含：任务拆解、技术架构、数据策略、执行阶段，以及**与 Cursor AI 的完整对话流程（可直接复制发送的 prompt 序列）**。

## 一、任务本质

本次笔试有**两条并行评分线**，必须同时照顾：

| 评分线 | 交付物 | 关键点 |
|---|---|---|
| 全栈 / 工程能力 | 爬虫代码 + 数据 + 分析 + HTML 报告 | 代码能跑、结构清晰、能处理反爬、数据→报告链路完整 |
| AI 协作能力 | 完整 Cursor transcript | 对话有逻辑、任务线清晰、体现「先规划再执行 / 分步验证 / 调试归因」 |

**核心洞察**：面试官重点看 transcript，所以「与 AI 的这段对话本身」就是交付物。发给 AI 的每条消息都要像一个靠谱工程师在驱动 AI——先拆需求、定方案、搭脚手架、逐模块实现、遇错调试、再分析出报告。

---

## 二、关键决策（已锁定）

1. **数据真实性**：真实爬取为主；任一平台被反爬挡住 → 降级到公开可查历史均价，并在报告 / README 中**明确标注来源与抓取时间**。（诚实标注 > 伪造漂亮假数据）
2. **技术深度**：允许使用 **Playwright**（自动化浏览器，处理登录 / JS 渲染 / 滑块）。房价站点优先用轻量 `httpx + parsel`，扛不住再切 Playwright。
3. **爬取时间范围**：由使用者在执行时告知 agent，或直接写入爬虫代码的时间段常量（默认覆盖近 3 年）。
4. **价格呈现**：使用**均价曲线**；额外**标注历史成交价最高 Top 10 与最低 Top 10 的时间节点**。
5. **天通苑粒度**：按我方思路（整体均价为主线，如数据允许可补充代表性子区对比）。
6. **报告叙事**：**双线并行**——宏观因果（政策事件 → 房价拐点归因） + 联动分析（微博/小红书舆情情绪 vs 房价走势）。
7. **事件清单**：覆盖 全国/北京层面（认房不认贷 2023.09、房贷利率多次下调、限购松绑、公积金调整）+ 天通苑本地（地铁 17 号线、旧改、学区、法拍房占比）。
8. **本机环境**：**未装过 Playwright** → 脚手架 / README 中必须包含 `pip install playwright && playwright install chromium` 步骤。

## 三、数据来源分工

| 数据类型 | 来源 | 用途 | 反爬难度 |
|---|---|---|---|
| 房价数值（成交/挂牌均价） | 链家/贝壳、安居客 | 报告主线，均价曲线 + 最高/最低 Top10 | 中 |
| 舆情热度 / 情绪 | 微博、小红书 | 联动分析（情绪 vs 房价） | 高（需登录/签名） |
| 政策与事件时间线 | 新闻 + 政策原文 | 宏观因果归因 | 结构化整理为主 |

> 说明：小红书/微博是「舆情来源」不是「房价数值来源」；房价数值以链家/安居客为准，并做交叉验证。

## 四、技术架构与目录结构

```
AutoEditGoods/
├── crawler/
│   ├── config.py            # UA 池 / 限速 / 时间段常量 / 天通苑小区 ID
│   ├── fetch_lianjia.py     # 链家成交/挂牌均价（房价核心数据）
│   ├── fetch_anjuke.py      # 安居客小区历史均价（交叉验证）
│   ├── fetch_weibo.py       # 微博舆情（关键词：天通苑/北京楼市/认房不认贷）
│   ├── fetch_xhs.py         # 小红书笔记（Playwright，小样本）
│   └── utils.py             # 指数退避重试 / 随机延时 / 存储
├── data/
│   ├── raw/                 # 原始 json / html
│   └── clean/               # price_monthly.csv / events.csv / sentiment.csv
├── analysis/
│   ├── clean_merge.py       # 清洗 + 按月聚合 + 事件/舆情时间对齐
│   └── attribution.py       # 涨跌归因（同比/环比 + 事件窗口 + Top10 极值）
├── report/
│   ├── build_report.py      # 生成自包含 HTML（内联 SVG，无外部依赖）
│   └── 天通苑房价分析报告.html   # ← 最终交付
├── requirements.txt
└── README.md                # 运行步骤 + 数据来源 + 抓取时间 + 局限声明
```

**技术选型**：Python 3 · `httpx`/`requests` + `parsel`/`BeautifulSoup`（静态页）· `Playwright`（登录/动态页）· `pandas`（清洗分析）· CSV/SQLite（存储）· 报告用 `Jinja2` 或纯 f-string 生成**内联 SVG**（对齐 demo 风格，保证离线可打开、无 CDN 依赖）。

## 五、执行阶段（6 阶段）

| 阶段 | 内容 | 产出 |
|---|---|---|
| 0 脚手架 | 目录 / requirements / config / utils（含重试与限速） | 可运行骨架 + Playwright 安装说明 |
| 1 房价爬虫 | 链家天通苑成交/挂牌均价，安居客交叉验证 | `data/raw/*` |
| 2 舆情+事件 | 微博关键词爬取 + 小红书小样本 + 政策事件结构化 | `sentiment` 原始数据 + `events.csv` |

## 六、与 AI 的对话流程（可直接复制的 prompt 序列）⭐

> 这是给面试官看的主线。按顺序发送；遇错时**不要删除重来**，而是把报错贴给 AI 让其归因修复——真实的「调试→定位→修复」回合是加分项。全程中文、任务线单一清晰。

**① 需求锁定 + 方案**（已完成本步）
> 让 AI 输出任务分析、执行方案、对话流程，并确认关键决策。

**② 搭脚手架**
> 「同意方案。先搭项目脚手架：创建目录结构、`requirements.txt`、`crawler/config.py`（含 UA 池、限速、爬取时间段常量、天通苑小区 ID）、`crawler/utils.py`（带指数退避重试和随机延时的请求封装）。README 里加上 Playwright 安装步骤（我没装过）。先不写具体爬虫。」

**③ 房价爬虫（核心）**
> 「实现 `fetch_lianjia.py`，爬取北京天通苑 [时间段] 的成交/挂牌均价，保存原始数据到 `data/raw/`。处理反爬（headers、限速、异常重试），先小规模试跑并把抓到的样例打印给我看。」

*（若被反爬挡住，发 ③.1）*
> 「链家返回了 [贴报错/验证码]。分析是什么反爬机制，给应对方案（换 headers / 加 Cookie / 上 Playwright / 降级安居客），选一个稳妥的实现。」

**④ 舆情 + 事件**
> 「实现 `fetch_weibo.py` 抓取关键词【天通苑、北京楼市、认房不认贷】近 3 年热门微博（小样本，带互动数和时间）；`fetch_xhs.py` 用 Playwright 抓小红书小样本；再整理近 3 年影响北京/天通苑房价的政策与事件，结构化成 `data/clean/events.csv`（字段：日期、事件、类型、方向[利好/利空]、说明、来源）。」

**⑤ 清洗合并**
> 「写 `analysis/clean_merge.py`：房价按月聚合成 `price_monthly.csv`，清洗异常值，找出成交价最高/最低 Top10 时间节点，对齐 events 和舆情时间轴。给我看每个表的前几行和缺失值情况。」

**⑥ 分析归因**
> 「写 `analysis/attribution.py`：计算 3 年趋势、同比/环比、主要涨跌拐点，把 events 对齐拐点做宏观归因；再做舆情情绪 vs 房价的联动分析。输出结论要点。」

**⑦ 生成报告**
> 「参考 `public/Manus事件专项报告 4.html` 的杂志风格和内联 SVG 图表方式，用真实数据生成 `report/天通苑房价分析报告.html`。章节：封面 → 数据概览 → 3 年均价走势折线(标注最高/最低 Top10) → 政策事件时间线 → 事件叠加价格图 → 舆情情绪联动图 → 涨跌归因结论 → 数据来源与局限。图表全部内联 SVG，离线可打开。」

**⑧ 收尾**
> 「写 README（运行步骤、数据来源、抓取时间、数据局限声明），并帮我 review 整个项目还有哪些面试会被追问的点。」

## 八、面试防追问清单

- **合规**：只抓公开数据、控制频率、注明来源、不做商用——主动说明体现职业素养。
- **反爬层次**：能讲清 UA / Cookie / 签名(x-s) / 验证码 / IP 限频的应对方案。
- **数据可信度**：链家 vs 安居客交叉验证、异常值处理、缺失月份插值说明。
- **归因严谨性**：强调「相关 ≠ 因果」，事件窗口仅为关联性分析。
- **工程化**：重试/限速/降级策略、配置与代码分离、数据分层（raw → clean）。
